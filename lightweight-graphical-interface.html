Lightweight Graphical Interface

Objectives, Requirements, Strategies

This paper represents thoughts about the objectives, requirements, and strategies I presented at the initial meeting of the SG13 Study Group in Chicago, 2013-Sept-28. These are my personal views and do not necessarily represent the views of other SG13 members.

Objective: Success

This paper isn't about the technical aspects of a library Technical Specification (TS) for lightweight graphical interface - rather it presents an approach to creating such a TS in a way that maximizes the chance of success by avoiding minefields.

Elephants in the room and differing expectations

There are major players in the problem domain - Microsoft, Apple, Open GL, QT, and the list goes on and on. Also, expectations differ widely as soon as we start talking about a graphical interface TS.

A strategy to defuse concern about undue competition and to calibrate expectations is to put "lightweight" or a synonym in the title of both the SG and the TS. Examples include "Concepts Lite" and some internal Boost components that use "Lightweight" in the title and have successfully avoided conflicts and limited expectations.

Avoiding design by committee

A lightweight graphical interface, even though far simpler that a full-fledged GUI framework, is far too complex to be successfully designed by a committee. Strategies are presented that taken together addresses this concern.

Strategy: Ship in two years

That implies a major feature complete working draft in two meeting. I.E. There is no time for design by committee.

Furthermore, there is a working implementation (on GitHub or similar) in sync with the working draft. That's the only way to have confidence that the WD is implementable and useful to users.

Strategy: WD starting point is an existing successful library already in wide use

Starting with an existing successful library that is already in wide use avoids design by committee and gives the SG and standards committee enough confidence that we can move forward quickly.

The strategy for accepting the STL into the standard is a useful role model. The whole STL proposal was put in the standard working draft, and only then were changes allowed.

Once SG13 has a working draft, we can accept changes of any magnitude based on their merits. If someone comes forward with a proposal to completely replace the library as specified in the WD, that proposal obviously would have to be very strong to make progress. Smaller proposals, particularly those that have been implemented (presumably as a branch of the GitHub implementation), might well be accepted, and small tweak proposals will certainly be accepted.\

Requirements for a starting point library

  • Existing successful library already in wide use.
     

  • Modern C++ that uses the C++ standard library where applicable, modulo naming conventions and C++11/14 features. (Dependencies hidden from users are exempt from this requirement.)
     

  • Extendible by third-parties. An initial library TS in a short time frame will be quite limited. That's much more acceptable if the library is extendible by third parties.
     

  • Owner willing to meet ISO intellectual property requirements. It would also be helpful if the original developers were willing to get involved in SG work.
     

  • Meets Beman's "Hello C++ World" challenge; displaying that text in a window and being able to drag it around inside the window is only slightly more complex that the traditional Hello world program.
     

  • Meets Herb's "Learn the library basics and build an interesting app in four hours" challenge.
     

  • There may be some other challenges or requirements, as long as they don't eliminate all candidates.

Does any existing library meet these requirements?
